## Agent 行为约束

### 自主执行原则（最重要）

**能 agent 做的，绝不让人工参与。** 具体边界：

| agent 自己做 | 必须让人做 |
|-------------|-----------|
| 跑命令、修报错、重启服务 | 看 GUI 窗口里的内容 |
| 读/写/改代码和配置文件 | 操作物理硬件或外设 |
| curl / 脚本批量测试 | 输入账号密码、扫码 |
| 安装依赖、修路径、加权限 | 判断"这个视觉效果对不对" |
| 报错自己诊断并修复后重跑 | 系统授权弹窗（辅助功能等） |

报错处理流程：**遇到错误 → 自己诊断 → 改代码/配置 → 重跑 → 直到通过，再汇报结果。** 不要遇到报错就抛给用户问"怎么办"。

启动命令（如 `pnpm tauri dev`、`uvicorn`、`cargo run`）也由 agent 在后台执行，日志写到 `/tmp/[项目名]-dev.log`，发现报错立即修复并重启，修干净后再通知用户。用户只在 **GUI 窗口出现后** 才需要介入（截图、点击、验收）。

人工操作 UI 期间，agent 实时监控日志（`tail /tmp/[项目名]-dev.log`），用户每做一步操作后 agent 立刻读日志定位问题，不等用户描述错误。定位到根因后自行修复，重启服务，再通知用户重试。

人工介入只在真正需要时发生，且 agent 要给出明确指令：「现在需要你做 X，做完告诉我结果」，一次只要求一个动作。

### 其他约束

- **读顺序**：先读此文件 → CONTEXT → DESIGN → PLAN，再动手
- 不修改 CONTEXT.md 架构决策部分，除非用户明确要求
- 不修改 DESIGN.md 功能点和 API 契约，除非用户明确要求
- 每个 Phase 完成后跑验收 checklist，**通过才进下一个**
- prompt 改动后必须跑 infra-model-eval 验证，不验证不提交
- `.env` 不提交 git
- 文档和代码同一个 commit 提交
